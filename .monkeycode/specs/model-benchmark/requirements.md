# 模型评测需求

## 介绍

模型评测用于针对同一模型名称，对多个可用端点执行一致的测试请求，比较可用性、首字节延迟、总耗时、输出质量和成本相关指标。模型评测与运行期延迟统计共享入口，但保留独立的评测任务、样本和结果。

## 术语

- **评测任务**：管理员创建的一次模型、提示词、候选端点和评测策略的组合。
- **候选端点**：声明支持目标模型并参与评测的端点。
- **评测样本**：评测任务内的一条固定输入提示词。
- **运行监控**：来自实际代理请求的端点延迟、成功率和错误率统计。
- **评审模型**：管理员选择的端点和模型，用于对被测输出生成效果评分。
- **效果评分**：评审模型依据评分标准生成的单样本输出评分。

## 需求

### 需求 1：创建评测任务

**用户故事：** 作为管理员，我希望选择一个模型和多个端点发起相同的评测，以便得到可比较的结果。

#### 验收标准

1. 当管理员创建评测任务时，系统应当要求管理员选择目标模型、至少两个候选端点和至少一条评测样本。
2. 当管理员启动评测任务时，系统应当向每个候选端点发送相同的模型参数和评测样本。
3. 当候选端点未声明目标模型时，系统应当在任务创建页面标记该候选端点的模型支持状态。
4. 当某个候选端点请求失败时，系统应当保留该候选端点的失败结果并继续执行其他候选端点。
5. 当管理员启动评测任务时，系统应当对每个候选端点和评测样本执行三次请求。
6. 当评测任务执行请求时，系统应当以串行方式执行候选端点和样本组合。

### 需求 2：收集性能结果

**用户故事：** 作为管理员，我希望查看每个端点的性能统计，以便选择响应稳定且速度合适的端点。

#### 验收标准

1. 当候选端点返回响应时，系统应当记录状态码、首字节延迟、总耗时、输出长度、Token 用量和错误信息。
2. 当评测任务完成时，系统应当按端点汇总成功率、平均首字节延迟、平均总耗时和 Token 用量。
3. 当管理员查看评测结果时，系统应当支持按性能指标排序并展示各端点的单样本明细。

### 需求 3：比较输出效果

**用户故事：** 作为管理员，我希望在相同提示词下并排比较模型输出，以便判断端点输出效果。

#### 验收标准

1. 当管理员打开某条评测样本时，系统应当并排展示每个候选端点的完整输出、失败信息和性能数据。
2. 当评测输出生成后，系统应当调用评审模型，依据管理员配置的评分标准为每个候选端点生成效果评分和评分理由。
3. 当评审模型调用失败时，系统应当保留被测端点的性能结果，并标记该评测结果的效果评分状态。
4. 当输出超过配置的保存阈值时，系统应当标记输出截断状态并保留已保存部分。
5. 当评审模型返回评分时，系统应当保存总分、准确性分、完整性分、指令遵循分、表达质量分、评分理由和置信度。
6. 当被测输出或评审响应超过 1MB 时，系统应当保存前 1MB 内容并标记截断状态。

### 需求 5：保留评测历史

**用户故事：** 作为管理员，我希望长期保存评测历史，以便跟踪端点表现变化。

#### 验收标准

1. 系统应当持久化保存已完成、失败和取消的评测任务及其结果。
2. 当服务重启后，系统应当恢复历史评测任务、尝试结果、评审结果和端点汇总。
3. 当管理员查看历史评测任务时，系统应当提供任务创建时的端点配置和模型配置快照。

### 需求 4：统一入口

**用户故事：** 作为管理员，我希望从统一入口了解实时运行表现和受控评测表现，以便基于不同数据来源做决策。

#### 验收标准

1. 系统应当提供一级功能“模型评测”。
2. 当管理员进入“模型评测”时，系统应当提供“评测任务”和“运行监控”两个页签。
3. 当管理员打开“运行监控”时，系统应当展示现有端点延迟排行榜和成功率指标。
4. 当管理员打开“评测任务”时，系统应当展示历史评测任务、状态和汇总结果。
